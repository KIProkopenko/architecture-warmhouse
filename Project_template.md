### ** Задания 1.**

### **Анализ файла**

Файл описывает проектную работу по переходу компании "Тёплый дом" от монолитного приложения к распределённой экосистеме умного дома. Вот ключевая информация:

*   **Текущее состояние (As-Is):**
    *   Монолитное приложение на **Go** с базой данных **PostgreSQL**.
    *   Синхронная архитектура без микросервисов.
    *   Функционал ограничен **управлением отоплением** и **мониторингом температуры**.
    *   Подключение требует выезда специалиста.
    *   Нет возможности самостоятельной регистрации устройств пользователями.
*   **Целевое состояние (To-Be):**
    *   Модель **SaaS** с самообслуживанием.
    *   Расширенный функционал: **освещение, ворота, камеры, и т.д.**.
    *   Возможность самостоятельного подключения и настройки устройств пользователями.
    *   Поддержка устройств партнёров.
*   **Цель задания:** Проанализировать монолит, выделить домены, и создать диаграмму контекста системы (C4) с помощью PlantUML.

---

### **Задание 1. Анализ и планирование**

#### **1. Функциональность монолитного приложения:**

*   **1.a. Управление отоплением:** Пользователь может через веб-интерфейс отправить команду на включение или выключение системы отопления в своём доме. Система передаёт эту команду через интернет на соответствующее оборудование.
*   **1.b. Мониторинг температуры:** Система периодически опрашивает датчики температуры, установленные в домах, и получает данные о текущем значении. Эти данные отображаются для пользователя в веб-интерфейсе.

#### **2. Архитектура монолитного приложения:**

*   **2.a. Язык программирования: Go** – Выбран для написания всего приложения.
*   **2.b. База данных: PostgreSQL** – Единая СУБД для хранения всех данных (пользователи, устройства, настройки, показания датчиков и т.п.).
*   **2.c. Архитектура: Монолитная** – Все компоненты (веб-сервер, бизнес-логика, работа с БД) собраны в одном исполняемом файле.
*   **2.d. Взаимодействие: Синхронное** – Операции выполняются последовательно. Запрос от пользователя блокирует обработку до завершения.
*   **2.e. Масштабируемость: Ограничена** – Для увеличения нагрузки нужно масштабировать весь монолит целиком, что неэффективно и дорого.
*   **2.f. Развертывание: Требует остановки всего приложения** – Любое изменение в одной части требует перезапуска всего приложения, что приводит к простою сервиса.

#### **3. Определение доменов и границ контекстов (DDD):**

Применение DDD помогает структурировать будущую систему. На основе целевой экосистемы можно выделить следующие основные **домены**:

*   **Core Domains (Ключевые домены):**
    *   **Управление устройствами (Device Management):** Отвечает за регистрацию, настройку, статус и команды, отправляемые конкретным устройствам.
    *   **Управление домом (Home Control):** Обеспечивает функции, связанные с контролем различных систем в доме (отопление, освещение, ворота), основываясь на данных из других доменов.
    *   **Мониторинг и телеметрия (Monitoring & Telemetry):** Сбор, хранение и предоставление данных с датчиков (температура, движение и т.д.).
*   **Supporting Domains (Вспомогательные домены):**
    *   **Пользовательская платформа (Customer Platform):** Р аутентификация, управление профилем, подписка SaaS.
    *   **Управление интеграциями (Integration Management):** Обеспечивает подключение и работу с API сторонних производителей устройств.
*   **Generic Domains (Общие домены):**
    *   **Уведомления (Notifications):** Отправка сообщений пользователям (например, о событиях безопасности).
    *   **Логирование и аудит (Logging & Audit):** Журнал действий и событий системы.

Границы контекстов (Bounded Contexts) будут соответствовать этим доменам, определяя, какие данные и функции принадлежат какому микросервису.

**Описание диаграммы:**

*   `Person(customer)`: Представляет пользователя системы – владельца умного дома.
*   `System_Boundary(system_boundary)`: Обозначает границы самой системы "Тёплый дом".
*   `System(monolith)`: Центральный элемент – текущее монолитное приложение.
*   `System(device_partner)`: Внешняя сущность – умные устройства, установленные в домах.
*   `System(internet)`: Представляет собой среду передачи данных (Интернет).
*   `Rel(...)`: Показывает отношения между сущностями (как кто с кем взаимодействует и по какому протоколу).

Эта диаграмма даёт общее представление о системе и её месте в окружающей среде.

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.
Locations - название комнаты, sensorId - идентификатор названия комнаты

	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081
